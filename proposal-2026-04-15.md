## 1. Executive Summary

ScrapeKit is a local-first, session-aware scraping CLI that automatically selects the optimal extraction strategy for static pages, JS-rendered SPAs, and authenticated sites — without requiring the user or AI agent to decide which method to use.

**What it does differently from existing tools:**
- Connects to the user's *already-running* Chrome, inheriting all login sessions and cookies. No re-authentication. No credential storage.
- For modern SSR frameworks (Next.js, Nuxt, SvelteKit), extracts data directly from the embedded JavaScript state variable — bypassing DOM rendering entirely.
- Caches the "extraction recipe" per site so subsequent runs skip discovery and go straight to the fastest path.

**Core claim:** The full strategy ladder (HTTP → CDP with existing session → JS state extraction) has been validated end-to-end on a production website. This is not a hypothetical design — the hardest parts already work.

**Target distribution:** Open source, published to PyPI (`pip install scrapekit`).

**MVP timeline:** 2–3 weeks for a solo developer with Python and browser automation experience.

## 2. Problem

Web scraping in 2026 is a tool selection problem before it is a data extraction problem.

A developer or AI agent facing a new URL must first answer:
- Is this a static page or a JS-rendered SPA?
- Does it require authentication?
- Is the data in the DOM, or embedded in a JavaScript variable?
- Should I use requests, Playwright, Scrapy, Crawl4AI, or something else?

This meta-decision adds friction for human developers and is particularly damaging for AI agents, which must resolve tool selection before beginning the actual task.

A second, underserved problem: **login-required pages**. Existing tools treat the browser as stateless — each scrape session launches a fresh browser, requiring re-authentication or cookie injection. Valuable data behind login walls (sports betting, financial platforms, member content) is effectively inaccessible to automated tools.

**A third discovery:** Many modern web frameworks (Next.js, Nuxt, SvelteKit) embed the full page dataset in a JavaScript variable (`window.__PRELOADED_STATE__`, `window.__NEXT_DATA__`, etc.) at page load time. This means the data is available *before* DOM rendering completes — but no existing scraping tool exploits this systematically.

---

## 3. Validated Insight (Origin of This Proposal)

This proposal originates from a working implementation built for scraping NBA betting odds from Sportsbet.com.au.

**What was discovered:**
Using HAR recording (capturing all network traffic during page load), the full odds dataset was found to be embedded in `window.__PRELOADED_STATE__` — a single JavaScript object containing all markets and outcomes, loaded before any DOM rendering.

**Result of switching from DOM scraping to state extraction:**
- Scrape time: 23 seconds → 6 seconds per page
- Data completeness: partial (DOM-visible only) → full (280+ markets)
- Stability: fragile (depends on DOM structure) → robust (fixed JSON path)
- 4 games scraped in parallel: ~8 seconds wall time

**Why this matters for ScrapeKit:**
This discovery pattern — HAR → find JS state variable → direct extraction — is not specific to Sportsbet. It applies to any site built on SSR frameworks. The mechanism is proven; what remains is generalising it into a reusable tool.

## 4. Technical Architecture

### 4.1 The Strategy Ladder

The core of ScrapeKit is an ordered escalation ladder. Each level is attempted in sequence; a level is only reached if the previous level failed or produced insufficient output.

```
Level 1 — HTTP Direct
  Tool:      httpx (async HTTP client)
  When:      Static HTML pages, public APIs, Wikipedia-type content
  Speed:     ~0.5–2 seconds
  Coverage:  ~30% of the open web
  Condition to escalate: Response body < 500 chars, or contains
             known SPA shell markers, or text-to-HTML ratio < 0.1

      ↓ escalate if Level 1 insufficient

Level 2 — CDP + Existing Chrome Session
  Tool:      Python Playwright (chromium.connect_over_cdp)
  When:      JS-rendered SPAs, login-required pages
  Speed:     ~5–10 seconds
  Coverage:  ~60% of the web (including most authenticated pages)
  Key advantage: Connects to the user's already-running Chrome.
             Inherits all cookies, sessions, and authentication.
             No re-login required. Single pip install, no Node.js.
  Condition to escalate: First visit with no cached profile, or
             profile indicates Level 3 needed

      ↓ escalate if Level 2 insufficient

Level 3 — JS State Extraction (+ HAR Discovery on first run)
  Tool:      Python Playwright (page.evaluate + HAR recording)
  When:      Data embedded in JS variables (Next.js, Nuxt, etc.)
  Speed:     First run ~15–20s (HAR discovery included);
             subsequent runs ~6s (cached profile, skip HAR)
  v0.1 scope: __PRELOADED_STATE__ only (one variable, fully proven)
  v0.2 scope: Generalise to __NEXT_DATA__, __NUXT__, __INITIAL_STATE__,
             plus auto-scan for unknown large JSON objects in <script>
  Output:    Site profile saved to cache; future requests skip
             directly to optimised extraction
```

### 4.3 Profile Cache

After a successful Level 3 extraction, the "recipe" for that domain is persisted locally:

```json
{
  "domain": "sportsbet.com.au",
  "level": 3,
  "data_source": "window.__PRELOADED_STATE__.entities.sportsbook",
  "discovered_at": "2026-04-15T22:00:00Z",
  "last_success": "2026-04-16T22:52:00Z",
  "success_count": 12,
  "failure_count": 0
}
```

On the next invocation for any URL on this domain, the Dispatcher reads this file and goes directly to Level 3 state extraction, bypassing HAR recording entirely.

**v0.1 cache scope:** Domain-level matching only. Simple, conservative, safe.  
**Cache invalidation:** Stale after 30 days, or immediately on extraction failure (triggers automatic re-discovery).

### 4.4 Dispatcher Escalation Logic

The two critical decision functions:

```python
def needs_browser(response: httpx.Response) -> bool:
    """Determine if HTTP response requires browser rendering."""
    if len(response.text) < 500:
        return True
    text_lower = response.text.lower()
    if any(s in text_lower for s in [
        "enable javascript", "you need javascript",
        "<div id=\"app\"></div>", "<div id=\"root\"></div>"
    ]):
        return True
    # Low text-to-HTML ratio indicates SPA shell
    soup = BeautifulSoup(response.text, "html.parser")
    text_ratio = len(soup.get_text()) / len(response.text)
    return text_ratio < 0.1

def needs_discovery(url: str) -> bool:
    """Determine if HAR discovery is needed."""
    if not profile_cache.exists(url):
        return True   # First visit to this domain
    if profile_cache.is_stale(url):
        return True   # Profile expired
    return False
```

> ⚠️ **`needs_browser()` is the highest-risk component in this proposal** (see Section 7). Its accuracy across a diverse set of real websites is the first thing a technical co-founder should validate before committing to the architecture.

## 5. Technology Stack

| Component | Technology | Rationale |
|-----------|-----------|-----------|
| Language | Python 3.11+ | Ecosystem fit, agent tooling |
| HTTP client | `httpx` | Async, HTTP/2, modern |
| HTML parsing | `beautifulsoup4` | Mature, widely understood |
| Browser / CDP | `playwright` (Python) | Native `connect_over_cdp()` support; single `pip install`; no Node.js |
| HAR recording | `playwright` (built-in) | `page.route` + response interception; same package |
| CLI framework | `typer` | Type-safe, auto-generated help |
| Profile storage | JSON files | Zero dependencies, human-readable |
| Output | JSON (v0.1), Markdown (v0.1) | Covers primary agent use cases |
| Distribution | PyPI | `pip install scrapekit` |

**Stack change from v0.1 proposal:**  
Node.js dependencies (`dev-browser`, `agent-browser`) have been eliminated. Python Playwright's `chromium.connect_over_cdp("http://localhost:9222")` natively connects to an already-running Chrome instance, providing identical functionality with no Node.js requirement.

**Installation experience:**
```bash
pip install scrapekit
playwright install chromium   # one-time, downloads ~120MB Chromium binary
scrape https://example.com
```

## 6. MVP Scope

### Phase 1 — Minimum viable but differentiated (v0.1)

The goal of Phase 1 is a complete end-to-end story: a user can run one command and get structured data from a static page, a logged-in SPA, *and* a Next.js-style site with embedded state — all without selecting a strategy manually.

```
scrape <url> [--output json|md] [--force-level 1|2|3]

  Level 1: HTTP + HTML extraction (httpx + beautifulsoup4)
  Level 2: CDP via Python Playwright (connect_over_cdp)
  Level 3: __PRELOADED_STATE__ extraction only
             (one variable, fully validated — no generalisation yet)

  Profile Cache: domain-level only, no management subcommands
  Output: JSON (default), Markdown
```

Level 3 is intentionally narrow in v0.1: only `window.__PRELOADED_STATE__`. This is the proven path. Expanding to other frameworks is Phase 2 work.

### Phase 2 — Generalisation and polish (v0.2)

```
  Level 3 expanded: __NEXT_DATA__, __NUXT__, __INITIAL_STATE__,
                    auto-scan <script> tags for large JSON objects
  HAR auto-discovery for unknown state variable names
  Profile management: scrape profile list|show|delete
  Output: CSV added
  PyPI packaging polish, README with live demo
  Basic test suite
```

### What both phases explicitly exclude

```
- Proxy / IP rotation
- Cloudflare / bot detection bypass
- Scheduled / cron scraping
- GraphQL query construction
- Web UI or dashboard
- Cloud deployment
- JavaScript interaction (clicking, form submission)
```

## 7. Feasibility Assessment

### What is proven (not hypothetical)

| Claim | Evidence |
|-------|---------|
| CDP connects to existing Chrome session | Running in production on Windows |
| HAR recording discovers JS-embedded state | Validated on Sportsbet.com.au |
| `__PRELOADED_STATE__` extraction works | 12 scrapes, structured JSON output |
| Speed: ~6s per page after profile caching | Measured, 4 parallel games in ~8s |
| Python Playwright supports `connect_over_cdp` | Official Playwright API, documented |

### What requires further validation

| Question | Risk Level | Notes |
|----------|-----------|-------|
| `needs_browser()` accuracy across diverse sites | **HIGH** | **This is the dispatch bottleneck. Getting it wrong (false positive OR false negative) directly degrades UX. Must be tested on 50+ URLs before v0.1 ships. This is the first thing a co-founder should validate.** |
| JS state variable coverage beyond __PRELOADED_STATE__ | Low | Well-documented framework patterns; Phase 2 concern |
| Profile Cache URL pattern matching accuracy | Low | Domain-level is conservative but safe for Phase 1 |
| Behaviour when Chrome is not running | Low | Clear error message; Level 1 still works without Chrome |
| Playwright CDP behaviour on Mac/Linux | Low | Official cross-platform support, but needs smoke testing |

### Competitive landscape

| Tool | Stars | Auto-strategy | Login session reuse | HAR/JS state extraction | Local-only |
|------|-------|---------------|--------------------|-----------------------|------------|
| Scrapy | ~52k | ❌ | ❌ | ❌ | ✅ |
| Playwright | ~70k | ❌ (library, not tool) | ❌ stateless | ❌ | ✅ |
| Scrapling | ~37k | ❌ user picks fetcher | ❌ | ❌ | ✅ |
| Crawl4AI | ~40k | ❌ | ❌ | ❌ | ✅ |
| Apify | — | Partial | ❌ | ❌ | ❌ cloud |
| **ScrapeKit** | — | ✅ | ✅ | ✅ | ✅ |

Scrapling (37k stars) is the closest open-source analogue. Its existence confirms real demand for adaptive scraping tools. Its gap — no auto-strategy selection, no session reuse, no JS state extraction — is exactly what ScrapeKit addresses.

## 8. Open Questions for Technical Reviewer

1. **`needs_browser()` accuracy (HIGH PRIORITY):** What is the false-positive/false-negative rate across 50+ diverse URLs? At what accuracy threshold is the heuristic "good enough" vs requiring an ML fallback? This should be the first experiment run before any other module is built.

2. **URL pattern matching granularity:** Domain-level cache matching is safe but coarse. `sportsbet.com.au/nba/*` and `sportsbet.com.au/afl/*` may eventually need different strategies. At what point does domain-level matching break down, and what is the right next step (regex patterns, URL prefix trees)?

3. **`__PRELOADED_STATE__` path generalisation:** The v0.1 implementation hard-codes extraction from `window.__PRELOADED_STATE__`. For Phase 2, how should the tool handle sites where the relevant data is nested at an unknown path within the state object? Is a user-provided JSON path (`--json-path entities.sportsbook`) acceptable, or must the tool infer it?

4. **Chrome availability assumption:** Both Level 2 and Level 3 require Chrome running locally with `--remote-debugging-port=9222`. This is documented as a requirement. Is this acceptable for the target user (AI agents, power users), or does the tool need a headless fallback that auto-launches a clean Chrome instance?

## 9. Suggested First Steps (If Moving Forward)

1. **Days 1–3:** Build CLI skeleton + HTTP Extractor. Immediately run `needs_browser()` against 50 diverse URLs (mix of static, SPA, login-required, Cloudflare-protected). Measure false-positive rate. **Do not proceed until this number is understood.**
2. **Days 4–6:** Add CDP Extractor via `playwright.connect_over_cdp`. Validate session reuse on 5 login-required sites using existing Chrome.
3. **Days 7–8:** Add `__PRELOADED_STATE__` extraction (Level 3 Phase 1 scope). Wire up Profile Cache at domain level.
4. **Days 9–10:** Integration pass, output formatter, basic README. Tag v0.1 and publish to PyPI (even as alpha).
5. **Phase 2 start:** Expand Level 3 to additional JS state variables. Write launch post — the Sportsbet 23s→6s story is the hook.
